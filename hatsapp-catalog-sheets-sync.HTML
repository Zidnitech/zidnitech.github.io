01 // Overview

This reference covers two systems wired together: the WhatsApp Catalog API, which lets customers browse and order products inside a chat thread, and a Google Sheets sync layer that keeps stock, pricing, and availability current without requiring a full database for smaller catalogs.

ARCHITECTURE
  ┌──────────────────────┐         ┌──────────────────────┐
  │   Google Sheet       │ ◄─────► │  Sync Worker (cron)  │
  │  (products, stock,   │  pull/  │  every 15 min         │
  │   prices, SKUs)      │  push   │                       │
  └──────────────────────┘         └──────────┬───────────┘
                                                │
                                       Meta Commerce Manager
                                       (Catalog Batch API)
                                                │
                                                ▼
  ┌─────────────────────────────────────────────────────────┐
  │              WhatsApp Catalog (customer-facing)          │
  │   Browse → Add to cart → Send order → Webhook fires      │
  └──────────────────────┬────────────────────────────────────┘
                          │
                          ▼
  ┌──────────────────────┐
  │  Order Webhook        │ ──► decrement stock in Sheet
  │  Handler              │ ──► notify staff / CRM
  └──────────────────────┘
Why Sheets instead of a database? For catalogs under ~2,000 SKUs, a Sheet gives non-technical staff direct edit access (update price, mark out-of-stock) without touching code or an admin panel. Migrate to a real DB once you need transactional consistency at higher volume.

02 // Catalog Setup (Meta Commerce Manager)

Before any code runs, the catalog must exist in Meta Commerce Manager and be connected to your WhatsApp Business Account (WABA).

Step 1 — Create the catalog

  1. Meta Commerce Manager → Create Catalog → type: E-commerce
  2. Connect catalog to your WABA under Business Settings → WhatsApp Accounts
  3. Note the catalog_id — needed for every API call below

Step 2 — Authenticate

ENV.env
META_ACCESS_TOKEN=your_system_user_token
CATALOG_ID=your_catalog_id
WABA_PHONE_NUMBER_ID=your_phone_number_id
GOOGLE_SHEET_ID=your_sheet_id
GOOGLE_SERVICE_ACCOUNT_JSON=./service-account.json

Step 3 — Upload products via Batch API

JSsrc/catalog/upload.js
const axios = require('axios');

/**
 * Push a batch of products to Meta's Catalog using the
 * Product Feed Batch API. Accepts up to 5,000 items per call.
 */
async function uploadProductBatch(products) {
  const requests = products.map(p => ({
    method: 'UPDATE',
    retailer_id: p.sku,
    data: {
      name: p.name,
      description: p.description,
      availability: p.stock > 0 ? 'in stock' : 'out of stock',
      condition: 'new',
      price: `${p.price} SAR`,
      currency: 'SAR',
      image_link: p.imageUrl,
      link: p.productUrl,
      brand: p.brand || 'Generic',
    }
  }));

  const { data } = await axios.post(
    `https://graph.facebook.com/v19.0/${process.env.CATALOG_ID}/batch`,
    { requests },
    { headers: { Authorization: `Bearer ${process.env.META_ACCESS_TOKEN}` } }
  );

  return data.handles; // track batch processing status
}

module.exports = { uploadProductBatch };
Required fields: retailer_id (your SKU, must be unique and stable — never reuse), price, availability, and image_link (must be a public HTTPS URL, no auth wall). Missing any of these silently rejects the item from sync.

03 // Sending Catalog Messages

Once products exist in the catalog, you can send them directly inside a WhatsApp conversation — either as a single product, a curated multi-product set, or the full browsable catalog.

Single Product Message

JSsrc/messages/sendProduct.js
async function sendSingleProduct(to, retailerId) {
  return sendMessage({
    to,
    type: 'interactive',
    interactive: {
      type: 'product',
      body: { text: 'هذا هو المنتج الذي سألت عنه 👇' },
      action: {
        catalog_id: process.env.CATALOG_ID,
        product_retailer_id: retailerId // e.g. "SKU-1042"
      }
    }
  });
}

Multi-Product Message (curated set)

async function sendProductSet(to, retailerIds, sectionTitle = 'مقترحاتنا لك') {
  return sendMessage({
    to,
    type: 'interactive',
    interactive: {

        sections: [{
          title: sectionTitle,
          product_items: retailerIds.map(id => ({ product_retailer_id: id }))

      }
    }
  });
}

04 // Google Sheets Sync Layer

The Sheet acts as the source of truth for stock and pricing. A scheduled worker reads it, diffs against the last known state, and pushes only the changed rows to Meta — keeping API calls minimal.

Expected Sheet Structure

ColumnFieldExample
AskuSKU-1042
Bnameسماعة لاسلكية X200
Cprice249
Dstock18
Eimage_urlhttps://...jpg
Fproduct_urlhttps://store.com/p/1042
Glast_syncedauto-filled by worker

Read + Diff + Push

JSsrc/sync/sheetSync.js
const { google } = require('googleapis');
const { uploadProductBatch } = require('../catalog/upload');

const auth = new google.auth.GoogleAuth({
  keyFile: process.env.GOOGLE_SERVICE_ACCOUNT_JSON,
  scopes: ['https://www.googleapis.com/auth/spreadsheets']
});

async function syncFromSheet() {
  const sheets = google.sheets({ version: 'v4', auth });

  const { data } = await sheets.spreadsheets.values.get({
    spreadsheetId: process.env.GOOGLE_SHEET_ID,
    range: 'Products!A2:F', // skip header row
  });

  const rows = data.values || [];
  const products = rows.map(r => ({
    sku: r[0],
    name: r[1],
    price: parseFloat(r[2]) || 0,
    stock: parseInt(r[3]) || 0,
    imageUrl: r[4],
    productUrl: r[5],
  }));

  // Only push rows that changed since last sync (cheap in-memory diff)
  const changed = await diffAgainstCache(products);
  if (changed.length === 0) return { updated: 0 };

  await uploadProductBatch(changed);
  await saveCache(products);

  return { updated: changed.length };
}

// Run every 15 minutes
const cron = require('node-cron');
cron.schedule('*/15 * * * *', syncFromSheet);
Why diff before pushing? Meta's Batch API has rate limits per app. Pushing all 2,000 rows every 15 minutes when only 3 changed wastes quota fast. The diff-cache pattern keeps you well under limits even on large catalogs.

05 // Order Webhooks (Cart → Sheet)

When a customer completes a cart inside WhatsApp, Meta sends an order message type to your webhook. This handler decrements stock back in the Sheet and notifies staff.

JSsrc/webhooks/order.js
async function handleOrderMessage(msg, contact) {
  const order = msg.order; // { catalog_id, product_items: [{product_retailer_id, quantity, item_price}] }

  const sheets = google.sheets({ version: 'v4', auth });
  const rowMap = await getSkuRowMap(); // sku -> row number, cached

  for (const item of order.product_items) {
    const row = rowMap[item.product_retailer_id];
    if (!row) continue;

    await sheets.spreadsheets.values.update({
      spreadsheetId: process.env.GOOGLE_SHEET_ID,
      range: `Products!D${row}`, // stock column
      valueInputOption: 'USER_ENTERED',
      requestBody: { values: [[`=D${row}-${item.quantity}`]] }
    });
  }

  await notifyStaff({
    customer: contact.profile.name,
    phone: contact.wa_id,
    items: order.product_items,
    total: order.product_items.reduce((sum, i) => sum + i.item_price * i.quantity, 0)
  });

  await sendOrderConfirmation(contact.wa_id, order);
}
Race condition note: Two simultaneous orders for the last unit of a low-stock item can both succeed before the Sheet updates. For high-demand SKUs, add a stock check + soft-lock before confirming the order, not just after.

06 // API Reference

POST/{catalog_id}/batch
Bulk create/update/delete catalog items. Max 5,000 requests per call. Returns a handles array for tracking async processing status.
GET/{catalog_id}/products
List all products in a catalog with pagination. Useful for periodic full-state reconciliation against the Sheet.
POST/{phone_number_id}/messages
Send catalog, product, or product_list interactive messages. Same endpoint used for all WhatsApp message types.
PUTProducts!D{row}
Sheets API value update — used internally by the order webhook handler to decrement stock by row reference.

07 // Choosing a Platform Layer

Everything above works directly against Meta's raw API. In practice, most teams put a managed platform (like Wati) in front of this for inbox management, agent handoff, and broadcast — while still using the catalog and Sheets code patterns above for the commerce logic.

Catalog sync from Sheets✅ Full control⚠️ Varies by provider
Shared team inbox❌ Build yourself✅ Built-in
Broadcast + templates UI❌ Manual via API✅ Visual builder
Per-message cost transparency✅ Direct Meta billing⚠️ Some add markup
